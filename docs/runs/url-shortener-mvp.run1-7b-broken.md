## Setup

The brief is identical across the three URL-shortener tier comparisons —
the same six-deliverable URL shortener CLI:

> Build a URL shortener CLI.
> - User can add a long URL and get a short 6-char hash back
> - can look up the original URL given the hash
> - can list all saved mappings
> - Persist to disk
> - Include tests for add/lookup/list plus collision handling

The DAG was generated by an earlier-sprint plan-builder pass (pre-C5
api_spec hardening, so the consumed plan had structural defects baked
in). Executor task list at runtime: `setup_project`,
`pytest_tests_contract`, `pytest_tests`, `core_domain_module`,
`cli_entry_point`, `pytest_tests_test_lookup`.

## What happened

The run took 62 minutes of wall-clock time and finished with 2 tasks
green and 9 failed (loopback-counted; 4 distinct task slots flipped to
`success=False`). Three patterns dominated the timeline:

1. **`add_function` mis-use loops.** The model repeatedly called
   `add_function` with multi-statement payloads, triggering the
   "single-function-only" guard. The auto-mark-complete synth captured a
   snapshot that gets attached to subsequent retries verbatim:

   > It seems there was an error because the `add_function` call included
   > more than one statement. Let's ensure that each call only includes a
   > single function definition. We need to make sure the `url_shor[…]

   The architect read this on the next turn and tried again, with the
   same multi-statement payload. The instructive part is that the
   feedback is *in the model's working memory* and it can't act on it.

2. **Pytest-routed retries.** Test failures upstream got routed back to
   the impl task per Agora's loopback machinery
   (`routing pytest error from task=pytest_tests_test_lookup upstream to
   task=core_domain_module (routed retry 1/2)`). The retry produced a
   slightly different impl that failed the same test for a slightly
   different reason. Two routed-retry rounds before loopback exhausted.

3. **Review-phase rejection by timeout.** With nine task slots failed,
   the review phase auto-rejected:

   > advance_phase: review -> failed (review rejected (loopback limit):
   > auto-rejected after timeout: 9 task(s) failed)

## What worked

- **The framework noticed.** `add_function` rejected multi-statement
  payloads. The pytest gate caught broken impls. Loopback routed retries
  upstream as designed. None of the failure modes ended in "shipped
  broken code that someone runs later"; they ended in "framework refused
  to advance the phase".
- **Auto-mark-complete synthesis.** Captured the model's textual
  rationale even when the model didn't call `mark_complete`, so the
  failure had a parseable narrative rather than vanishing.

## What broke

- **The 7B model could not absorb its own feedback.** The classic
  capability-ceiling pattern (memory note
  `project_model_capability_ceiling.md`): same hallucination across
  retries, even with the corrective text in the system prompt.
- **62 minutes of GPU time for 2 successful tasks.** Wall-clock cost is
  the real expense at this tier. The model wasn't cheap; it just wasn't
  billed per token.

## What changed in the framework as a result

This run *itself* did not drive a framework change — the runs that drove
fixes were the discord-bot Runs 1–13 documented in lessons-learned.md. What
this run *did* do was establish the **capability-ceiling baseline for
the cross-tier comparison**. Sections 4.1 and 4.4 of [findings.md](findings.md)
read against this run.

It is also the canonical "here is what 7B does on a six-deliverable
brief without staging" data point. The Run 17 reference
(`discord-bot-full.live`) on the same model showed that *with*
template-driven staging on every Python-producing task, the same 7B
hits 23/23 first-pass. The contrast is the load-bearing argument that
the framework's leverage on weak models comes from staging + literal
templates, not from postconditions alone.
